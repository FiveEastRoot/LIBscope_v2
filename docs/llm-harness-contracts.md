# LIBscope LLM 하네스 계약 문서

기준일: 2026-06-13  
관리 코드: `functions/_shared/llm-output-contracts.cjs`  
현재 계약 버전: `llm-output-contracts-v0.1`  
분석 신호 문서: `docs/llm-analysis-harness-guide.md`

## 목적

이 문서는 실제 LLM 연결 전에 고정해야 하는 입력, 출력, 문체, 근거, 검수, 캐시, 보고서 구조를 정의한다. 모델은 이 계약 안에서만 동작하며, 모델이나 provider가 바뀌어도 화면과 보고서의 양식이 흔들리지 않도록 한다.

수치 계산, 평균 대비, 분위, 순위, 데이터 계보는 LLM이 생성하지 않는다. 코드가 먼저 `analysisSignals`를 만들고 LLM은 이 산출물을 해석한다.

## 1. 생성 단위

| 생성 단위 | section key | 성격 | 기본 저장 방식 |
| --- | --- | --- | --- |
| 자치구 종합 인사이트 | `districtInsight` | 화면 최상단 3카드 요약 | 자치구 snapshot 기준 캐시 |
| 인구구조 분석 | `population` | 변동성 지표 해석 | 자치구 snapshot 기준 캐시 |
| 문화역량·향유 지표 | `culture` | 고정 데이터셋 사전 해석 | 최초 생성 후 DB 저장 |
| 교육인프라 | `education` | 학교·도서관 연계 해석 | 자치구 snapshot 기준 캐시 |
| 사회안전망 대상자 구성 | `socialSafety` | 복지·장애·외국인 접근성 해석 | 자치구 snapshot 기준 캐시 |
| 주변 공공기관·문화시설 | `publicPlaces` | 도서관 입지 주변 2km 해석 | 도서관/좌표 snapshot 기준 캐시 |
| 자치구 보고서 다운로드 | `districtReport` | HTML/Markdown 보고서 | 보고서 버전 아카이브 |

## 2. 공통 출력 슬롯

모든 섹션 해석은 자유문 한 덩어리로 받지 않는다. 기본 출력은 다음 슬롯을 사용한다.

| 슬롯 | 목적 | 제한 |
| --- | --- | --- |
| `title` | 섹션 제목 | 1개 문자열 |
| `summary` | 화면 또는 보고서 앞부분에 쓰는 요약 | 2-3문장 |
| `keyFindings` | 핵심 근거와 해석 | 2-4개 |
| `cautions` | 해석 유의사항 | 최대 2개 |
| `evidenceRefs` | 입력 지표 참조 | 1-8개 |
| `qualityFlags` | 검수 상태/주의 플래그 | 필요 시 배열 |
| `recommendedView` | 화면 표현 방식 제안 | 선택 |
| `reportUse` | 보고서에서 사용할 위치/문장 | 선택 |
| `analysisBasis` | 코드가 계산한 비교 기준과 데이터 계보 | 선택 |

`districtInsight`는 `summary` 대신 3개의 `cards`를 사용한다. `districtReport`는 `sections`, `markdown`, `html`을 추가로 사용한다.

## 3. 섹션별 입력·출력 계약

### 3.1 자치구 종합 인사이트

- 입력: 인구, 문화, 교육, 사회안전망, 주변 공공기관 요약 패킷
- 출력: `cards` 3개, `cautions`, `evidenceRefs`, `qualityFlags`
- 문장 제한: 카드당 1문장 또는 짧은 판단문 1개
- 화면 표시: 카드당 핵심 불릿 2-4개
- 카드 라벨: `핵심 판단`, `주의 지점`, `실행 방향`
- 해석 규칙:
  - 최소 3개 지표축을 함께 반영
  - 다양한 섹션의 부분 인사이트를 짧은 불릿으로 압축
  - 화면 카드에는 `명`, `개`, `건`, `%` 등 수치를 직접 나열하지 않음
  - 각 카드에는 자치구 고유 조건 또는 키워드를 2개 이상 포함
  - 어느 자치구에도 그대로 붙일 수 있는 범용 지표 목록형 문장 금지
  - 정책 처방으로 과잉 확장 금지
  - 자치구 보고서의 결론 흐름과 충돌 금지

### 3.2 인구구조 분석

- 입력: 주민등록인구, 생활인구, 연령대, 성별, 출처
- 출력: 공통 출력 슬롯
- 문장 제한: `summary` 2문장, `keyFindings` 최대 4개
- 해석 규칙:
  - 주민등록인구와 생활인구를 같은 증감 지표처럼 직접 비교하지 않음
  - 0-9세, 10-64세, 65세 이상 구간을 서비스 대상 구분으로 해석
  - 생활인구 결측 또는 fallback 상태를 숨기지 않음

### 3.3 문화역량·향유 지표

- 입력: 2023 서울문화지표, 2024 서울시민 문화향유 참고값
- 출력: 공통 출력 슬롯
- 문장 제한: `summary` 2문장, `keyFindings` 최대 4개
- 해석 규칙:
  - 문화향유 값은 서울시 집단별 참고값으로 표현
  - 자치구 직접 순위처럼 쓰지 않음
  - 고정 데이터셋은 최초 생성 후 DB 저장

### 3.4 교육인프라

- 입력: 학교급별 개수, 학교명, 주소, 도서관 프로그램/행사 정보
- 출력: 공통 출력 슬롯
- 해석 규칙:
  - 학교 수를 도서관 수요의 직접 추정치로 단정하지 않음
  - 학교급별 규모와 실제 위치 목록을 분리
  - 도서관·학교·문화시설 연계 가능성은 실행 처방으로 표현

### 3.5 사회안전망 대상자 구성

- 입력: 수급자, 가구 구성, 장애 대분류/상세, 외국인 주민 유형, 국적 구성
- 출력: 공통 출력 슬롯
- 해석 규칙:
  - 대상자 집단을 지역 문제의 원인처럼 표현하지 않음
  - 가구, 장애, 외국인 축을 분리
  - 접근성, 언어, 이동, 정보 도달성 관점으로 해석

### 3.6 주변 공공기관·문화시설

- 입력: 선택 도서관 좌표, 2km 이내 기관 목록, 거리, 기관 유형
- 출력: 공통 출력 슬롯
- 해석 규칙:
  - 거리와 기관 유형을 중심으로 입지 협력 가능성 해석
  - 2km 이내 목록은 실제 접근성을 보장하지 않음을 명시
  - 문화시설, 행정기관, 복지기관, 교육기관을 유형별로 분리

### 3.7 자치구 보고서 다운로드

- 입력: 모든 섹션 해석 패킷, 자치구 종합 인사이트, 시각화 사양, 보고서 템플릿 버전
- 출력: `title`, `subtitle`, `sections`, `markdown`, `html`, `evidenceRefs`, `qualityFlags`
- 해석 규칙:
  - 보고서 목차와 섹션 순서는 템플릿이 통제
  - 모델은 슬롯만 채움
  - 화면용 종합 인사이트와 결론 흐름이 충돌하지 않음
  - 시각화는 지표명, 단위, 기준시점을 유지

## 4. 문체 규칙

- 존대 표현 사용 금지
- 명사형 또는 보고서형 종결 우선
- "필요합니다", "보입니다", "확인됩니다" 대신 "필요", "해석 가능", "확인" 사용
- "좋다/나쁘다"보다 "검토 필요", "연계 가능", "주의 필요" 사용
- 대상 집단을 문제 원인처럼 표현 금지
- 정책 원인은 단정하지 않되, 데이터 해석은 실행 방향으로 연결

## 5. 근거 표기 규칙

수치 기반 문장은 가능한 한 다음 정보를 함께 유지한다.

- 지표명
- 값
- 단위
- 기준연도 또는 기준월
- 출처 또는 source key
- 고정 데이터/변동 데이터 여부

고정 데이터와 갱신형 데이터는 다음처럼 분리한다.

- 고정 데이터: PDF, 수동 추출 CSV, 조사 참고값. 원자료 교체 전까지 재집계하지 않음.
- 갱신형 데이터: API 또는 Supabase 캐시 기반 값. `reference_date`, snapshot, cache version 변경 시 재계산.
- 혼합 판단: 고정값과 갱신값을 함께 쓰는 경우 추세나 인과관계가 아니라 교차 검토 질문으로 표현.

예시:

- 허용: "65세 이상 인구 41,538명, 주민등록인구 기준 고령층 비중 확인."
- 지양: "고령층이 매우 많아 복지 수요가 크다."

## 6. 품질 게이트

| 게이트 | 실패 조건 |
| --- | --- |
| JSON schema 검증 | 필수 슬롯 누락, 배열 길이 초과, 잘못된 타입 반환 |
| 근거 수치 검증 | 입력에 없는 수치 생성, 지표명 누락, 단위 누락, 기준연도/월 누락 |
| 문체 검증 | 존대형 종결 반복, 화면 카드 문장 과다, 명사형 어미 위반 |
| 민감 지표 표현 검증 | 대상 집단을 문제 원인처럼 표현, 취약성 과잉 단정, 근거 없는 정책 처방 |
| 섹션 간 일관성 검증 | 화면 인사이트와 보고서 결론 충돌, 고정/API 데이터 기준 혼합 |
| 캐시 가능성 검증 | `snapshot_key`, `harness_version`, `model_registry_version`, `prompt_version` 누락 |
| 분석 신호 검증 | `analysisSignals` 누락, 코드 산출 비교값이 아닌 모델 생성 순위 사용 |

실패 시 처리 순서:

1. 동일 모델로 1회 재시도
2. 동일 prompt를 상위 모델로 승격
3. 실패 결과와 입력 payload를 저장
4. 수동 검토 대상으로 표시
