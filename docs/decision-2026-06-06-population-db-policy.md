# Decision: Population, Welfare Rate, Foreigner Metric, and DB Policy

작성일: 2026-06-06

## 결정 요약

- 인구 지표는 주민등록인구를 기본값으로 사용합니다.
- 생활인구는 병행 제공하며, 시각화 창 우상단 전환 버튼으로 표시 데이터를 바꿉니다.
- 주민등록인구와 생활인구는 둘 다 사전 로드하고, 전환 시에는 이미 로드된 데이터 중 보이는 값만 바꿉니다.
- 수급률 산식의 분모는 주민등록인구로 고정합니다.
- 외국인 지표는 외국인 주민 기준으로 정의합니다.
- 웹DB는 Supabase Postgres 기준으로 설계하되, 무료 플랜 한도 안에서 운영 가능한 구조로 제한합니다.

## 배경

LIBscope v2는 공공도서관 정책 의사결정을 위한 자치구/개별도서관 단위 인사이트 대시보드입니다. 기존 파일 기반 통계를 API 또는 주기 갱신 백엔드로 전환하는 과정에서, 인구 기준과 산식 기준을 명확히 고정해야 지표 해석의 일관성을 유지할 수 있습니다.

## 설계 영향

- 프런트엔드는 인구 시각화 컴포넌트에 `resident`와 `living` 데이터셋을 모두 받을 수 있어야 합니다.
- API 응답은 기본 주민등록인구와 병행 생활인구를 같은 응답 안에 담거나, 화면 진입 시 두 요청을 병렬로 수행할 수 있어야 합니다.
- 수급률 계산 로직은 `recipients / resident_population` 기준으로 고정합니다.
- 외국인 관련 API 매핑은 등록외국인보다 외국인 주민 데이터셋을 우선 검토합니다.
- Supabase에는 지표별 기준 인구, 출처, 기준일, 갱신 시각을 함께 저장해야 합니다.

## Supabase 무료 플랜 운영 원칙

- 주 1회 갱신을 기본으로 하고, 실시간성이 필요한 문화행사/Kakao Local 계열만 짧은 캐시 또는 직접 API를 유지합니다.
- 지표 테이블은 넓은 테이블보다 `metric_key + metric_json` 중심의 재사용 가능한 구조로 시작합니다.
- 읽기 트래픽이 작다는 전제에서 materialized summary 또는 사전 집계 JSON을 저장해 함수 응답 비용을 낮춥니다.
- 원천 API 응답 전체를 무기한 저장하지 않고, 서비스에 필요한 집계 결과와 출처 메타데이터만 저장합니다.

## 후속 작업

- 주민등록인구 API 후보의 실제 필드와 행정동 코드 매핑을 검증합니다.
- API 응답 스키마에 `populationModes` 또는 동등한 구조를 설계합니다.
- 인구 시각화 UI 우상단에 전환 버튼을 추가합니다.
- 수급률 계산 로직과 출처 표기를 주민등록인구 기준으로 수정합니다.
- Supabase migration SQL 초안을 작성합니다.
