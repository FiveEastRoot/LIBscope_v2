# LIBscope LLM 하네스 기획 문서 v0.1

작성일: 2026-06-09
대상: LIBscope v2 자치구 판단 단위 LLM 적용 설계
상태: 기획 초안

## 1. 문서 목적

이 문서는 LIBscope v2의 지표 묶음별 판단 단위에 LLM을 어떤 방식으로 연결할지 정의하는 기획 초안이다.

목표는 단순한 지표 설명 자동화가 아니라, API와 DB, 고정 지표, 보고서 생산, 문서 관리 절차를 연결하는 소규모 하네스를 구성하는 것이다. 하네스는 LLM 모델이나 버전이 변경되어도 원하는 방향의 판단 결과가 유지되도록 입력, 출력, 처리 규칙, 금지 표현, 검수 기준, 저장 전략을 고정한다.

향후 LIBscope는 지역사회 분야 인사이트 팟으로 기능하는 것을 목표로 한다. 자치구 공공데이터뿐 아니라 도서관 자체 DB나 운영 데이터 파일을 받았을 때도, 지역사회 분석과 도서관 정책 의사결정에 도움이 되는 해석과 보고서를 생산할 수 있어야 한다.

## 2. 공통 운영 원칙

### 2.1 LLM 적용 기준

LLM은 다음 조건 중 하나 이상에 해당하는 경우에 붙인다.

- API 기반 또는 변동성 있는 데이터의 해석이 필요한 경우
- 여러 지표 묶음을 결합해 판단해야 하는 경우
- 사용자가 보고서, 요약, 실행 제안처럼 문장형 산출물을 요구하는 경우
- 고정 데이터셋이라도 최초 기준 해석을 대량 생성해 DB에 저장할 필요가 있는 경우
- 도서관 운영 데이터처럼 원자료를 받아 의미 있는 정책 판단 단위로 변환해야 하는 경우

LLM을 붙이지 않아도 되는 경우는 다음과 같다.

- 단일 숫자, 단순 목록, 정렬, 합계, 비율처럼 코드로 명확히 처리 가능한 경우
- 시각화 자체가 충분히 해석 가능한 경우
- 지표 정의나 출처 표기처럼 고정 문구가 더 정확한 경우

### 2.2 하네스 문서의 역할

각 판단 단위별 문서는 단순 프롬프트 모음이 아니다. 다음 역할을 함께 수행한다.

- 입력 데이터 스키마 정의
- 출력 양식 정의
- 분석 관점과 우선순위 정의
- 금지 표현과 과잉 해석 방지 규칙 정의
- 캐싱과 저장 기준 정의
- 보고서 파이프라인에서의 위치 정의
- 향후 학계와 실무자 의견 반영 시 문서 목적성이 변질되지 않도록 변경 관리 기준 제공

### 2.3 문서 관리 방식

권장 관리 구조는 다음과 같다.

- Obsidian: 원본 하네스 문서 편집 및 버전 관리
- Notion: 검토용 기획문서와 공유용 문서 허브
- Supabase: 생성 결과, 입력 스냅샷, 프롬프트 버전, 모델 버전, 생성 시점 저장
- Netlify Functions: LLM 호출, 데이터 압축, 캐싱, 권한 관리

각 하네스 문서는 다음 메타데이터를 가져야 한다.

- 문서 ID
- 판단 단위명
- 적용 화면
- 입력 스키마 버전
- 출력 스키마 버전
- 프롬프트 버전
- 검수 상태
- 마지막 수정일
- 변경 사유
- 책임자 또는 검토자

### 2.4 모델 변경 대응 원칙

모델 또는 버전이 바뀌어도 결과 방향이 유지되도록 다음을 고정한다.

- 입력 필드명과 의미
- 출력 섹션명과 순서
- 문장 수 또는 분량
- 어조와 문체
- 종결어미 규칙
- 금지 표현
- 판단 우선순위
- 근거 지표 표기 방식
- 모호할 때의 보수적 표현 방식

모델 변경 시에는 대표 자치구 3~5개를 기준으로 골든 테스트를 수행한다.

골든 테스트 후보:

- 강남구: 인구 규모와 교육/문화 인프라가 큰 자치구
- 종로구 또는 중구: 생활인구와 문화자원이 강한 도심 자치구
- 노원구 또는 은평구: 생활권 기반 도서관 정책 해석이 중요한 자치구
- 금천구 또는 도봉구: 인프라와 복지 수요 균형을 봐야 하는 자치구

#### 비용 대비 성능 라우팅 원칙

모든 LLM 호출에 최고가 모델을 기본 적용하지 않는다. LIBscope의 기본 운영 목표는 충분한 인사이트 품질을 유지하면서 반복 호출 비용을 통제하는 것이다.

기본 라우팅은 다음과 같다.

- 단문 지표 해석과 고정 데이터 사전 생성: 저비용 모델 우선
- 자치구 종합인사이트: mini/flash급 균형 모델 우선, 품질 게이트 실패 시 승격
- 자치구 보고서 다운로드: 중상급 모델 기본, 기관 제출본 또는 민감한 결론 검토 시 고품질 모델 승격
- 문화역량처럼 고정 데이터셋 기반 출력: 최초 생성 후 DB 저장, 운영 중 반복 호출 금지
- provider 후보: Netlify AI Gateway 경로의 OpenAI/Gemini/Anthropic 우선, Mistral은 별도 직접 API 기반 저비용 후보로 분리

모델 자체의 카탈로그와 교체 절차는 `docs/llm-model-registry.md`를 기준으로 관리하고, 비용 라우팅과 운영 판단은 `docs/llm-model-recommendations.md`를 기준으로 관리한다. 섹션별 입출력, 문체, 근거, 품질 게이트, 보고서 목차는 `docs/llm-harness-contracts.md`를 기준으로 관리한다. 런타임 코드에서는 `functions/_shared/llm-model-registry.cjs`와 `functions/_shared/llm-output-contracts.cjs`를 기준으로 사용한다.

### 2.5 문체 및 종결어미 원칙

모든 보고서와 텍스트 생성물은 존대 표현보다 보고서 목적에 적합한 명사형 또는 보고서형 종결을 기본으로 한다. 이는 화면용 짧은 인사이트, 지표 묶음별 해석 패킷, 자치구 보고서, 향후 도서관 운영데이터 분석 보고서에 모두 적용한다.

기본 원칙:

- 존대형 종결인 "필요합니다", "보입니다", "확인됩니다", "권장됩니다"를 기본값으로 사용하지 않는다.
- 명사형 또는 보고서형 종결인 "필요", "가능", "확인", "검토 필요", "연계 가능성", "우선 고려"를 기본 사용한다.
- 사용자의 행동을 직접 지시하는 문장보다 판단 결과를 정리하는 문장을 우선한다.
- 화면용 인사이트는 짧은 명사형 문장 또는 간결한 서술형을 사용한다.
- 보고서 본문은 공공기관 보고서 문체를 따르되, 과도하게 고리타분한 한자어와 관성적 행정문구는 피한다.
- 문장 끝이 모두 같은 명사형으로 반복되어 부자연스러울 경우, 제한적으로 평서형을 섞을 수 있으나 존대형은 사용하지 않는다.

권장 예시:

- "고령층 비중과 1인가구 규모를 함께 고려한 생활밀착형 서비스 검토 필요"
- "생활인구 유입이 큰 지역으로 방문·체류 수요 기반 프로그램 연계 가능"
- "문화시설 공급은 충분하나 장애인 문화향유 접근성은 별도 확인 필요"

비권장 예시:

- "생활밀착형 서비스를 검토해야 합니다."
- "방문 수요가 높은 것으로 보입니다."
- "접근성 개선이 필요합니다."

## 3. 추천 생산 파이프라인

### 3.1 핵심 제안

자치구 종합인사이트와 자치구 보고서의 선후 관계는 다음 구조가 가장 자연스럽다.

먼저 각 지표 묶음별로 표준화된 해석 패킷을 생성한다. 이 해석 패킷이 공통 근거 레이어가 된다. 그 다음 최상단 종합인사이트와 자치구 보고서는 같은 해석 패킷을 사용하되, 출력 분량과 목적만 다르게 가져간다.

즉, 종합인사이트를 먼저 만들고 보고서가 그것을 따라가는 구조도 아니고, 전체 보고서를 먼저 만든 뒤 요약만 떼어내는 구조도 아니다. 중간 산출물인 자치구 해석 패킷을 먼저 만들고, 종합인사이트와 보고서가 이를 각각 소비한다.

### 3.2 처리 흐름

1. 원천 지표 수집
2. 지표 묶음별 데이터 정규화
3. 데이터 스냅샷 키 생성
4. 지표 묶음별 LLM 해석 패킷 생성 또는 캐시 조회
5. 최상단 종합인사이트 생성
6. 필요 시 자치구 보고서 생성
7. HTML, Markdown, PDF 변환
8. Supabase 저장 및 아카이빙
9. Notion 또는 Obsidian 문서 관리 흐름과 연결

### 3.3 데이터 스냅샷 키

보고서와 해석 결과는 단순히 자치구명으로 캐싱하지 않는다. 다음 값을 조합해 스냅샷 키를 만든다.

- 자치구
- 지표 기준연도 또는 기준일
- 데이터 소스 버전
- DB 적재 시점
- 하네스 문서 버전
- 출력 스키마 버전

예시:

```text
district:gangnam|metrics:2024-20260604|source:kosis+seoulapi+csv|harness:v0.1|schema:v0.1
```

## 4. 판단 단위 1: 자치구 종합인사이트

### 4.1 목적

자치구 종합인사이트는 사용자가 대시보드에 진입했을 때 선택 지역의 핵심 판단 방향을 짧게 파악하도록 돕는 영역이다. 전체 보고서의 축약판이 아니라, 지표 묶음별 해석 패킷을 바탕으로 현재 화면에서 가장 먼저 봐야 할 판단 포인트를 압축한다.

### 4.2 입력

입력은 원천 지표 전체가 아니라, 아래 항목을 압축한 자치구 해석 패킷을 우선 사용한다.

- 인구구조 해석 패킷
- 문화역량·향유 해석 패킷
- 교육인프라 해석 패킷
- 사회안전망 해석 패킷
- 주요 원천 지표 요약값
- 데이터 기준일과 출처

필수 입력 필드:

- district_name
- population_summary
- welfare_summary
- culture_summary
- education_summary
- social_safety_summary
- source_dates
- unavailable_metrics

### 4.3 출력 양식

현재 UI 구조를 유지한다.

- 핵심 요약: 1문장
- 주의 지표: 1문장
- 검토 방향: 1문장

각 문장은 80자 내외를 권장한다. 화면 카드 안에서 줄바꿈이 과도하게 발생하지 않도록 한다.

### 4.4 LLM 처리 규칙

- 숫자를 나열하지 말고 지표 간 관계를 요약한다.
- 가장 큰 수치가 아니라 정책 판단에 영향을 주는 조합을 우선한다.
- 고령층, 장애인, 외국인 주민, 수급자, 아동·청소년 관련 언급은 낙인 표현을 피한다.
- "위험하다", "심각하다", "취약하다" 같은 단정적 표현은 원자료가 명확히 뒷받침할 때만 사용한다.
- 불확실한 경우 "확인할 필요가 있습니다", "함께 검토할 수 있습니다"처럼 보수적으로 표현한다.
- 특정 사업을 바로 처방하기보다 보고서나 상세 섹션으로 이어지는 판단 방향을 제시한다.

### 4.5 저장 전략

종합인사이트는 생성 비용이 낮지만 화면 진입 시 반복 호출될 수 있다. 지표 스냅샷 키와 하네스 버전을 기준으로 Supabase에 저장한다. 지표 갱신 전까지는 저장값을 우선 표시한다.

### 4.6 검수 기준

- 3개 카드가 서로 중복되지 않는가
- 특정 지표 하나만 과도하게 강조하지 않는가
- 보고서의 판단 흐름과 충돌하지 않는가
- 현재 화면의 지표로 근거를 추적할 수 있는가

## 5. 판단 단위 2: 자치구 보고서 다운로드

### 5.1 목적

자치구 보고서는 모든 지표 묶음에 대한 LLM 해석이 완료된 뒤, 정해진 판단 절차와 문서 양식에 따라 생성되는 공식 보고서형 산출물이다. 목적은 화면에서 보이는 단편 해석을 넘어 정책 검토, 회의자료, 내부 공유 문서로 사용할 수 있는 결과물을 만드는 것이다.

### 5.2 입력

보고서 생성 시 입력은 다음 순서를 따른다.

1. 자치구 원천 지표 데이터
2. 지표 묶음별 해석 패킷
3. 지표 기준일과 출처
4. 시각화용 데이터셋
5. 보고서 템플릿 버전
6. 시각화 규칙 문서 버전
7. 하네스 판단 절차 버전

보고서 LLM은 원천 지표만 받아서 즉흥적으로 판단하지 않는다. 먼저 지표 묶음별 하네스가 생성한 해석 패킷을 읽고, 그 흐름과 어긋나지 않는 방식으로 종합한다.

### 5.3 출력 형식

기본 산출물은 HTML이다.

HTML을 기준으로 다음 파생 출력이 가능해야 한다.

- PDF 다운로드
- Markdown 변환
- Notion 업로드용 Markdown
- 내부 검토용 HTML 미리보기

문체는 공기업 보고서 느낌을 유지하되, 고리타분하거나 촌스러운 표현은 피한다. 존대형보다 명사형 또는 보고서형 종결을 기본으로 한다. 문장은 명료하고, 근거와 판단이 분리되어야 한다. 과도한 수식어나 홍보성 표현은 금지한다.

### 5.4 보고서 기본 구성안

분량은 아직 확정하지 않는다. 초기 권장안은 6~10쪽 분량의 웹 보고서 또는 A4 기준 4~6쪽 PDF다.

구성 초안:

1. 표지 및 보고서 개요
2. 핵심 요약
3. 데이터 기준과 출처
4. 자치구 인구구조 분석
5. 문화역량·향유 분석
6. 교육인프라 분석
7. 사회안전망 대상자 구성 분석
8. 지표 간 종합 판단
9. 도서관 정책 검토 방향
10. 실행 후보와 후속 확인 사항
11. 부록: 지표 정의와 시각화

### 5.5 시각화 규칙

시각화는 웹사이트 디자인을 그대로 복제하는 것이 목표가 아니다. 보고서 독자가 이해하기 쉽고 왜곡이 없는 방식으로 구성한다.

별도 문서로 관리할 항목:

- 색상 사용 규칙
- 축과 단위 표기 규칙
- 비율과 명수 병기 규칙
- 100% 누적 차트 사용 조건
- 소수점 자리수
- 누락 데이터 표기
- 비교 기준선 표기
- PDF 변환 시 해상도 기준
- 차트 제목과 출처 표기 방식

### 5.6 저장 전략

보고서는 토큰 비용이 크므로 생성 결과를 반드시 저장한다.

권장 저장 단위:

- district_report_id
- district_name
- data_snapshot_key
- harness_version
- report_template_version
- model_name
- model_version
- generated_at
- html_body
- markdown_body
- pdf_file_url 또는 storage_path
- source_metric_hash
- status

지표 갱신 시점 전까지는 DB 저장 보고서를 우선 보여준다. 지표 갱신 이후 다시 생성 요청이 들어오면 새 보고서를 생성하고 기존 보고서는 아카이빙한다. 연간 추이나 연도별 변화 분석을 하기 전까지는 아카이빙만 수행한다.

### 5.7 추가 논의 필요 사항

- 지표 갱신 시점 정의
- 보고서 재생성 트리거
- PDF 생성 라이브러리
- Supabase Storage 사용 여부
- 보고서 승인 상태 관리
- 사용자가 수동으로 보고서 재생성을 요청할 수 있는지 여부
- 보고서 분량 기준

## 6. 판단 단위 3: 인구구조 분석

### 6.1 목적

인구구조 분석은 주민등록인구와 생활인구를 바탕으로 지역 내 서비스 수요의 기본 구조를 해석한다. 화면에서는 짧은 해석문을 제공하고, 보고서에서는 더 자세한 판단 근거로 사용한다.

### 6.2 입력

필수 입력:

- 주민등록인구 총합
- 생활인구 총합
- 연령대별 인구
- 성별 인구
- 0~9세, 10~64세, 65세 이상, 70세 이상 구간
- 기준일
- 데이터 소스
- 누락 행정동 여부

선택 입력:

- 자치구 평균 또는 서울 평균
- 도서관 수
- 사회안전망 지표
- 교육기관 지표

### 6.3 출력 양식

화면용 짧은 해석:

- 핵심 구조: 1문장
- 서비스 고려점: 1문장
- 데이터 주의점: 필요 시 1문장

보고서용 해석:

- 인구 규모와 구조
- 생활인구와 주민등록인구 차이
- 연령 구간별 서비스 수요
- 도서관 정책과의 연결

### 6.4 분석 관점 가이드라인

- 65세 이상과 70세 이상은 고령층 서비스 접근성과 연결한다.
- 0~9세와 학령기 인구는 교육인프라와 함께 해석한다.
- 생활인구가 주민등록인구보다 크면 방문·통근·체류 수요 가능성을 언급한다.
- 생활인구가 unavailable이면 이를 해석하지 않고 데이터 주의점으로만 표기한다.
- 성별 비율은 차이가 정책적으로 의미 있을 때만 언급한다.
- 단순히 "많다/적다"가 아니라 어떤 서비스 설계에 영향을 줄 수 있는지로 연결한다.

### 6.5 금지 규칙

- 인구 감소나 고령화를 위기처럼 단정하지 않는다.
- 성별 비율만으로 프로그램 선호를 추정하지 않는다.
- 생활인구를 실제 거주 인구처럼 표현하지 않는다.
- 기준일이 다른 지표를 직접 비교할 때는 기준 차이를 표기한다.

## 7. 판단 단위 4: 문화역량·향유 지표 해석

### 7.1 목적

문화역량·향유 지표는 자치구의 문화 기반과 시민 문화향유 참고값을 연결해 도서관의 문화 접근성 역할을 해석하는 판단 단위다.

이 지표는 대부분 고정 데이터셋이므로 최초 생성 시 자치구별 해석 결과를 DB에 저장하고, 이후 화면에서는 저장값을 우선 조회한다.

### 7.2 입력

필수 입력:

- 공공문화시설 수
- 인구 10만 명당 공공문화시설
- 도서관 수
- 인구 10만 명당 도서관
- 공공공연장
- 공연시설
- 전시시설
- 생활문화시설
- 무장애 인증 문화공간
- 2024 서울시민 문화향유 참고값
- 일반시민, 문화관심층, 장애인 구분

### 7.3 출력 양식

사전 생성 화면용 해석:

- 문화 기반 해석: 1~2문장
- 도서관 연결 가능성: 1문장
- 접근성 고려점: 필요 시 1문장

보고서용 해석:

- 시설 공급 기반
- 향유 참고값과의 연결
- 장애인 또는 문화관심층 관점
- 도서관 문화 프로그램 방향

### 7.4 저장 전략

고정 데이터셋이므로 자치구별 해석 결과를 미리 생성해 Supabase에 저장한다.

저장 필드:

- district_name
- metric_group: culture_enjoyment
- source_year
- harness_version
- generated_text_short
- generated_text_report
- source_metric_json
- generated_at

보고서 생성 시에는 저장된 해석문과 원천 지표를 함께 입력한다. 저장 해석문은 판단 흐름의 기준선이며, 보고서 LLM은 다른 지표 묶음과의 관계를 반영해 문장을 조정할 수 있다.

### 7.5 분석 관점 가이드라인

- 문화시설 수만으로 문화 접근성이 충분하다고 판단하지 않는다.
- 인구 대비 지표와 절대 수를 함께 본다.
- 장애인 문화향유 참고값은 접근성 해석에 연결한다.
- 도서관을 단순 시설 수가 아니라 생활권 문화접근 거점으로 해석한다.
- 문화관심층 지표는 잠재 협력 또는 프로그램 확장 가능성으로 본다.

## 8. 판단 단위 5: 교육인프라

### 8.1 목적

교육인프라 해석은 자치구 내 초·중·고·대학 인프라를 기반으로 도서관의 아동·청소년, 학습, 진로, 지역교육 협력 가능성을 판단하는 영역이다.

### 8.2 입력

필수 입력:

- 초등학교 수와 목록
- 중학교 수와 목록
- 고등학교 수와 목록
- 대학교 수와 목록
- 학교명
- 주소
- 데이터 출처

선택 입력:

- 0~9세 인구
- 10대 인구 또는 학령기 인구
- 도서관 수
- 문화시설
- 주변 공공기관

### 8.3 출력 양식

화면용 해석:

- 교육 연계 가능성: 1문장
- 대상층 힌트: 1문장
- 협력 방향: 필요 시 1문장

보고서용 해석:

- 학교급별 분포
- 학령기 인구와의 연결
- 도서관 교육 프로그램 방향
- 대학·평생학습 협력 가능성

### 8.4 분석 관점 가이드라인

- 학교 수가 많다고 교육 수요가 자동으로 높다고 단정하지 않는다.
- 학령기 인구와 함께 해석한다.
- 초등학교는 어린이·보호자·기초문해 프로그램과 연결한다.
- 중·고등학교는 진로, 미디어 리터러시, 학습지원, 청소년 공간과 연결한다.
- 대학교는 평생학습, 전문강좌, 지역협력, 연구·봉사 연계 가능성으로 본다.
- 학교명 목록은 협력 후보 탐색의 재료이지, 개별 학교 평가의 근거가 아니다.

### 8.5 금지 규칙

- 학교별 수준이나 성과를 추정하지 않는다.
- 특정 학교를 우열로 평가하지 않는다.
- 학생 수 데이터가 없을 때 학교 수만으로 프로그램 규모를 제안하지 않는다.

## 9. 판단 단위 6: 사회안전망 대상자 구성 분석

### 9.1 목적

사회안전망 대상자 구성 분석은 가구, 장애, 외국인 주민 관련 지표를 통해 서비스 접근성, 정보 접근, 프로그램 설계 시 우선 고려할 수요 특성을 파악하는 영역이다.

### 9.2 입력

필수 입력:

- 가구 구성
- 장애 대분류 구성
- 외국인 주민 유형
- 등록외국인 국적 구성
- 기초생활 수급률
- 기준일
- 데이터 출처

선택 입력:

- 고령층 비중
- 1인가구 수
- 문화향유 장애인 참고값
- 무장애 인증 문화공간
- 주변 복지기관
- 공공기관 목록

### 9.3 출력 양식

화면용 해석:

- 수요 특성: 1문장
- 서비스 접근성 고려점: 1문장
- 프로그램 설계 힌트: 필요 시 1문장

보고서용 해석:

- 가구 구성 해석
- 장애 대분류 구성 해석
- 외국인 주민 유형 및 국적 구성 해석
- 복지·문화·도서관 서비스 연결
- 우선 고려 대상과 후속 확인 사항

### 9.4 분석 관점 가이드라인

- 1인가구는 고립 가능성으로 단정하지 않고, 생활정보·커뮤니티·접근성 설계 관점에서 본다.
- 장애 구성은 서비스 접근성, 보조기술, 무장애 환경, 정보 접근성과 연결한다.
- 외국인 주민 유형은 언어 지원, 생활정보, 다문화 프로그램, 지역 정착 지원과 연결한다.
- 국적 구성은 문화 다양성의 참고값으로 쓰되, 특정 국적에 대한 고정관념을 만들지 않는다.
- 기초생활 수급률은 서비스 비용, 접근성, 정보전달 경로와 함께 해석한다.
- 복지 수요는 결핍의 언어보다 지원 설계의 언어로 표현한다.

### 9.5 금지 규칙

- 대상자를 문제 집단처럼 표현하지 않는다.
- 장애, 국적, 복지 수급 여부를 낙인화하지 않는다.
- 외국인 국적 데이터만으로 문화 선호나 행동을 추정하지 않는다.
- 단순 비율만으로 서비스 우선순위를 확정하지 않는다.

## 10. 개별 도서관 파트 보류

- 지표 갱신 시점 정의
- 하네스 문서의 Notion/Obsidian 동기화 방식
- 최초 LLM 적용 범위: 종합인사이트부터 할지, 문화역량 고정 해석부터 할지

## 13. 현재 추천 결론

현재 단계에서는 자치구 종합인사이트를 바로 LLM에 붙이기보다, 먼저 지표 묶음별 해석 패킷 구조를 정의하는 것이 좋다. 그 이유는 종합인사이트와 보고서가 같은 판단 근거를 공유해야 하기 때문이다.

실제 구현 우선순위는 다음이 적절하다.

1. 문화역량·향유 고정 해석을 자치구별로 사전 생성해 DB 저장
2. 인구구조, 교육인프라, 사회안전망 해석 패킷 스키마 확정
3. 자치구 종합인사이트를 해석 패킷 기반으로 생성
4. 자치구 보고서 HTML 템플릿 설계
5. 보고서 생성과 캐싱 구현

이 순서가 토큰 비용을 줄이고, 판단 흐름의 일관성을 유지하며, 나중에 실무자와 학계 의견을 반영하기에도 가장 관리하기 쉽다.
